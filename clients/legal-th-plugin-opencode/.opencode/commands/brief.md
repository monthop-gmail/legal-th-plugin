# /brief -- สรุปข้อมูลกฎหมาย

> หากเห็น placeholder ที่ไม่คุ้นเคย ดู [CONNECTORS.md](../CONNECTORS.md)

สรุปข้อมูลกฎหมายตามบริบท รองรับ 3 โหมด: สรุปรายวัน, ค้นคว้าหัวข้อ, สรุปเหตุการณ์เร่งด่วน

**สำคัญ**: คำสั่งนี้ช่วยรวบรวมข้อมูล ไม่ใช่คำปรึกษาทางกฎหมาย

## Invocation

```
/brief daily                    # สรุปรายวัน
/brief topic [หัวข้อ]            # ค้นคว้าหัวข้อเฉพาะ
/brief incident [เหตุการณ์]      # สรุปเหตุการณ์เร่งด่วน
```

หากไม่ระบุโหมด ถามผู้ใช้ว่าต้องการแบบไหน

## Daily Brief -- สรุปรายวัน

สรุปสิ่งที่ทีมกฎหมายต้องรู้เพื่อเริ่มต้นวัน

### แหล่งข้อมูลที่ตรวจสอบ

**Email (ถ้าเชื่อม ~~email):**
- คำขอรีวิวสัญญาใหม่
- คำถาม compliance
- ตอบกลับจากคู่สัญญา
- จดหมายจากที่ปรึกษาภายนอก

**Calendar (ถ้าเชื่อม ~~calendar):**
- ประชุมที่ต้องเตรียมข้อมูลกฎหมาย
- กำหนดส่งสัญญา/เอกสาร
- deadline ที่ใกล้ถึง

**Chat (ถ้าเชื่อม ~~chat):**
- ข้อความในช่องทีมกฎหมาย
- ข้อความที่ tag ทีมกฎหมาย
- คำขอเร่งด่วน

**กฎหมาย (ผ่าน ~~legal-th):**
- ข่าวสารกฎหมายล่าสุด
- ประกาศ/ระเบียบใหม่
- กำหนดบังคับใช้ที่ใกล้ถึง

### Output Format

```
## สรุปรายวัน -- [วันที่]

### เร่งด่วน / ต้องดำเนินการ
[รายการที่ต้องจัดการทันที เรียงตามความเร่งด่วน]

### สัญญา Pipeline
- **รอรีวิว**: [จำนวนและรายการ]
- **รอคู่สัญญาตอบ**: [จำนวนและรายการ]
- **ใกล้ deadline**: [รายการที่ต้องเสร็จสัปดาห์นี้]

### คำขอใหม่
[คำขอรีวิวสัญญา NDA คำถาม compliance ที่เข้ามาตั้งแต่สรุปครั้งก่อน]

### ประชุมวันนี้
[ประชุมที่เกี่ยวข้องกับกฎหมาย + สิ่งที่ต้องเตรียม]

### ข่าวสารกฎหมาย
[กฎหมายใหม่ ประกาศ ระเบียบที่เกี่ยวข้อง]

### deadline สัปดาห์นี้
[กำหนดส่งที่ใกล้ถึง]

### แหล่งที่ไม่ได้ตรวจ
[แหล่งข้อมูลที่ไม่ได้เชื่อมต่อ]
```

## Topic Brief -- ค้นคว้าหัวข้อ

ค้นคว้าและสรุปข้อมูลเกี่ยวกับหัวข้อกฎหมายเฉพาะ

### Workflow

1. รับหัวข้อจากผู้ใช้
2. ค้นหาข้อมูลจากแหล่งที่เชื่อมต่อ:
   - **~~legal-th**: กฎหมาย ระเบียบ คำพิพากษาที่เกี่ยวข้อง
   - **เอกสาร**: บันทึก วิเคราะห์ ความเห็นก่อนหน้า
   - **Email**: การสื่อสารเรื่องที่เกี่ยวข้อง
3. สังเคราะห์เป็นสรุป

### Output Format

```
## สรุปหัวข้อ: [หัวข้อ]

### สรุปย่อ
[2-3 ประโยค สรุปภาพรวม]

### ภูมิหลัง
[บริบทและประวัติจากแหล่งข้อมูล]

### กฎหมายที่เกี่ยวข้อง
[กฎหมาย มาตรา ระเบียบที่เกี่ยวข้อง]

### สถานะปัจจุบัน
[จุดยืน/แนวทางปัจจุบันขององค์กร]

### ประเด็นสำคัญ
[ปัจจัย ความเสี่ยง คำถามที่ยังเปิดอยู่]

### ข้อมูลที่ยังขาด
[สิ่งที่ยังไม่พบ แหล่งที่ไม่ได้ตรวจ]

### ขั้นตอนถัดไป
[สิ่งที่ควรทำต่อ]
```

## Incident Brief -- สรุปเหตุการณ์เร่งด่วน

สรุปเหตุการณ์ที่ต้องการความสนใจทางกฎหมายทันที (ข้อมูลรั่วไหล, ถูกฟ้อง, หน่วยงานสอบสวน ฯลฯ)

### Workflow

1. รับรายละเอียดเหตุการณ์
2. สแกนแหล่งข้อมูลทั้งหมดอย่างรวดเร็ว:
   - **Email/Chat**: การสื่อสารเกี่ยวกับเหตุการณ์
   - **เอกสาร**: นโยบาย แผนรับมือ ประกัน
   - **~~legal-th**: กฎหมายที่เกี่ยวข้อง ระยะเวลาแจ้ง
3. รวบรวมเป็น brief ที่ดำเนินการได้ทันที

### Output Format

```
## สรุปเหตุการณ์: [เหตุการณ์]
**จัดทำเมื่อ**: [วันเวลา]
**ระดับความรุนแรง**: [ประเมินเบื้องต้น]

### สรุปสถานการณ์
[สิ่งที่ทราบเกี่ยวกับเหตุการณ์]

### Timeline
[ลำดับเหตุการณ์ตามเวลา]

### ข้อพิจารณาทางกฎหมายเร่งด่วน
[ระยะเวลาแจ้งหน่วยงาน, การเก็บรักษาหลักฐาน, attorney-client privilege]
- **PDPA**: แจ้ง สคส. ภายใน 72 ชม. หากเป็นเหตุละเมิดข้อมูล
- **กฎหมายอื่น**: [ตามแต่เหตุการณ์]

### สัญญาที่เกี่ยวข้อง
[สัญญา ประกัน ข้อตกลงที่อาจเกี่ยวข้อง]

### การตอบสนองภายใน
[กิจกรรมรับมือที่เกิดขึ้นแล้ว]

### ผู้ติดต่อสำคัญ
[บุคคลภายในและภายนอกที่เกี่ยวข้อง]

### สิ่งที่ต้องทำทันที
1. [เร่งด่วนที่สุด]
2. [ลำดับที่สอง]
3. [ฯลฯ]

### ข้อมูลที่ยังขาด
[สิ่งที่ยังไม่ทราบและต้องหาคำตอบ]
```

## หมายเหตุทั่วไป

- หากแหล่งข้อมูลไม่พร้อมใช้ แจ้งให้ชัดเจนว่าอะไรไม่ได้ตรวจ
- Brief ทุกประเภทต้อง actionable: ทุกรายการต้องมีขั้นตอนถัดไปชัดเจน
- กระชับ ลิงก์ไปยังเอกสารต้นฉบับแทนการคัดลอกทั้งหมด
- Incident brief ต้องเร็ว ออกข้อมูลที่มีก่อน ไม่ต้องรอครบ
